## Objetivo (1 frase)

"Ao fim deste sprint, o **Admin consegue gerir promotores e usuários** por uma tela protegida, a **Marina se cadastra com perfil completo e aguarda aprovação**, o sistema **autoriza por papel de verdade**, e o **promotor pendente ou rejeitado usa a plataforma como participante** enquanto aguarda — destravando toda a vertente-promotor das próximas sprints e fechando as dívidas ADR-T01/T02/T03."

## Histórias selecionadas

| ID | História (Como \<ator\> quero \<ação\> para \<valor\>) | Critérios de aceite operacionais |
|---|---|---|
| **US-051** | Como **sistema**, quero que o papel do usuário trafegue no token JWT e no header `X-User-Papel` para que cada serviço consiga autorizar por papel de verdade e fechar a dívida ADR-T01 | 1. Token JWT de um usuário com papel ADMIN, decodificado manualmente (ex.: jwt.io), contém o claim `papel: "ADMIN"`. 2. Token de participante tem `papel: "PARTICIPANTE"`. 3. Token gerado antes desta sprint (sem claim `papel`) continua válido e é tratado como PARTICIPANTE — o sistema não quebra. 4. **Bruno (participante) autenticado tenta acessar `GET /api/users`** (endpoint de admin) e recebe **403 "Acesso restrito a administradores."** — confirmado via Postman/curl. 5. **Bruno tenta navegar para `/admin/usuarios` no frontend** — é redirecionado imediatamente (não vê a tela e não vê o item "Admin" no menu). 6. Após qualquer login bem-sucedido, o gateway encaminha o header `X-User-Papel` ao serviço downstream (verificável em log do user-service). |
| **US-052** | Como **Marina (promotora)**, quero me cadastrar informando meu perfil completo (CPF, telefone, e-mail de contato, endereço com CEP, logradouro, número, complemento, bairro, cidade e UF; e redes sociais: Instagram e website) para que o admin tenha todas as informações necessárias para avaliar e aprovar minha conta | 1. Marina abre a tela de cadastro, escolhe a aba "Promotor" e **preenche todos os campos obrigatórios** (nome, e-mail, senha, CPF, telefone, e-mail de contato, CEP, logradouro, número, bairro, cidade, UF) sem precisar rolar mais de uma seção de cada vez — a aba está organizada nas seções visuais *Acesso*, *Identificação*, *Contato*, *Endereço*, *Redes*. 2. Ao tentar submeter com **CPF em branco**, o formulário marca o campo e exibe mensagem de erro **antes** de enviar ao servidor. 3. CPF é exibido com máscara `000.000.000-00`; CEP com `00000-000`; telefone com `(00) 00000-0000` — Marina nunca precisa digitar traço, ponto ou parêntese. 4. Cadastro bem-sucedido exibe confirmação: **"Cadastro enviado! Aguarde a aprovação do administrador."** e status fica PENDENTE. 5. **Cenário de concorrência — CPF duplicado:** dois cadastros simultâneos com o mesmo CPF → exatamente um retorna 201; o segundo retorna **409 "CPF já cadastrado"** visível no formulário (não tela de erro genérica). 6. Dois cadastros com o mesmo e-mail → idem: 409 "E-mail já cadastrado". 7. Cadastro de promotor com UF inválida (minúscula ou 3 caracteres) → 400 com campo apontado. 8. Instagram e website são opcionais — omiti-los não impede o envio. |
| **US-050** | Como **Admin**, quero uma tela de controle de usuários onde consigo ver todos os usuários, filtrar por papel e status, aprovar/rejeitar promotores pendentes (com motivo obrigatório na rejeição) e ver o **perfil completo** do promotor, para exercer a governança da plataforma | 1. Admin loga com a conta seed (`admin@pegaticket.local`) e vê o item **"Admin"** no menu de navegação — um participante não vê o item. 2. A tela `/admin/usuarios` exibe tabela paginada com nome, e-mail, papel e status de cada usuário. 3. Admin filtra por **"PROMOTOR / PENDENTE"** e vê apenas os cadastros que aguardam aprovação. 4. Admin clica no detalhe de um promotor pendente e vê o **perfil completo** (CPF, telefone, e-mail de contato, endereço, redes sociais) em drawer lateral. 5. **Admin aprova um promotor pendente em ≤ 2 cliques** (ação na linha + confirmação ou ação direta) → o status muda imediatamente para **VERIFICADO** na tela sem recarregar a página. 6. Admin rejeita um promotor informando o motivo (campo obrigatório no modal) → status muda para **REJEITADO** e o motivo fica registrado visível no detalhe. 7. **Idempotência:** Admin clica "Aprovar" duas vezes seguidas (duplo clique acidental) → ação processada uma vez; a tela não exibe erro 500 e o status segue VERIFICADO. 8. Participante autenticado chama `PUT /api/users/{id}/aprovar` via Postman → recebe **403 "Acesso restrito a administradores."**. |
| **US-053** | Como **Admin**, quero ativar ou inativar qualquer usuário para bloquear contas indevidas e controlar o acesso à plataforma sem precisar excluí-las | 1. Na tela `/admin/usuarios`, Admin clica **"Inativar"** na linha de um usuário ativo → o status muda visualmente para INATIVO na tabela sem recarregar a página. 2. **Usuário inativado tenta logar** → recebe **403** com a mensagem exata **"Conta inativa. Entre em contato com o administrador."** — não aparece "credenciais inválidas" nem erro genérico. 3. Admin reativa o usuário → o usuário consegue logar normalmente em seguida. 4. Admin tenta inativar a própria conta seed → o sistema **bloqueia a ação** com mensagem "Você não pode inativar sua própria conta." 5. Token JWT emitido antes da inativação segue válido pelo tempo restante (até 1h) — comportamento aceito no MVP; a tela de administração exibe aviso explicativo. 6. **Marina inativada, com token ainda válido, tenta acessar endpoint autenticado** → a requisição é aceita até o token expirar; no login seguinte, recebe 403 "Conta inativa" (limitação do MVP registrada). |
| **US-054** | Como **Marina (promotora avaliada)**, quero receber e-mail informando a decisão (aprovação com boas-vindas ou rejeição com o motivo), poder usar a plataforma como participante enquanto aguardo, e poder **reenviar minha solicitação** se for rejeitada, para não ficar travada e ter chance de corrigir minha candidatura | 1. **Papel base:** Marina se cadastra como promotora e, enquanto aguarda decisão, **já consegue logar** e usa a plataforma como participante (vê a Home, vê eventos futuros) — não fica presa em tela de bloqueio. 2. **Promotor rejeitado também usa como participante:** Marina, com status REJEITADO, **loga normalmente** e acessa a plataforma como participante — nenhuma tela a impede de navegar. 3. **E-mail de aprovação:** ao Admin clicar "Aprovar", Marina recebe e-mail de aprovação (assunto e corpo reconhecíveis) **visível no MailHog em dev** em ≤ 30 s. O e-mail contém o nome de Marina e link para o painel. O papel de Marina muda para PROMOTOR e `verificado=true`. 4. **E-mail de rejeição com motivo:** ao Admin rejeitar informando o motivo, Marina recebe e-mail de rejeição **com o motivo exato escrito pelo admin** — visível no MailHog em dev. Marina permanece com papel PARTICIPANTE e pode continuar usando a plataforma. 5. **Falha de envio de e-mail não derruba a transação:** se o servidor de e-mail estiver indisponível durante aprovação/rejeição, a operação do admin é concluída normalmente e apenas um log de erro é registrado — o usuário admin não recebe 500. 6. **Reenvio pós-rejeição:** Marina, estando REJEITADA, acessa a seção "Seja Promotor" (ou endpoint `POST /api/promotores/solicitar`) e reenvia a solicitação com os dados corrigidos → seu status volta para **PENDENTE** e o motivo de rejeição anterior é limpo. A Admin consegue ver a nova solicitação na fila de pendentes. 7. **Badge no front:** enquanto PENDENTE, Marina vê no seu perfil/nav: "Participante · solicitação em análise" (não "Promotor (pendente)"). Após aprovação, passa a ver "Promotor". |

## Atores exercitados

- **Bruno (participante, 24):** cenário de **acesso negado** — testa que toda a camada de admin está invisível e bloqueada para ele (menu, tela `/admin/usuarios` e todos os endpoints `/api/users/*`). Garante que o Sprint 1 não amplia a superfície de ataque para participantes. Bruno também representa o estado base de Marina enquanto ela aguarda aprovação como promotora.
- **Marina (promotora, 35):** cenário principal do sprint — **cadastro completo** com formulário guiado por seções e máscaras, espera digna pelo resultado da aprovação, recebimento de e-mail de aprovação ou rejeição com motivo, uso contínuo da plataforma como participante enquanto pendente ou rejeitada, e capacidade de **reenviar a solicitação** após rejeição. Marina é a sujeita passiva das ações do Admin.
- **Admin (operação):** cenário central — **governa promotores e usuários**: visualiza pendentes com perfil completo no drawer, **aprova/rejeita em ≤ 2 cliques**, inativa contas problemáticas, e recebe confirmação visual imediata de cada ação. Fluxo rápido, sem burocracia.

## Riscos de produto

| Risco | Impacto no ator | Probabilidade | Mitigação |
|---|---|---|---|
| Formulário de promotor com muitos campos (5 seções) → Marina abandona antes de enviar | Alta para Marina (taxa de conversão) | Média | Seções colapsáveis ou wizard de etapas; campos opcionais (Instagram, website) claramente sinalizados; validação inline antes do submit |
| Inativação não derruba sessão ativa (token stateless, TTL 1h) | Admin espera bloqueio imediato, mas o token ainda vale por até 1h | Alta | Aceito no MVP; aviso explícito na tela de administração junto ao botão "Inativar"; blacklist/refresh token é sprint futura |
| Seed de admin com senha hard-coded no repo (dev) | Baixo em dev; crítico se vazar para prod | Alta | Documentado como dev-only; tornar credencial env-driven antes de qualquer deploy real (dívida ADR-T05) |
| Falha no envio de e-mail silencia a decisão do admin (Marina não sabe que foi aprovada) | Médio para Marina e para Admin | Baixa | Falha de e-mail só loga (não reverte transação); painel de Marina deve exibir o status atual para que ela não dependa exclusivamente do e-mail |
| Admin aprova promotor por engano — não há "desfazer" direto | Médio para Admin | Baixa | Rejeitar após aprovar deve funcionar (idempotência inversa); fluxo clarificado na UI |
| Escopo da tela admin maior que o planejado (filtros + drawer + ações + paginação) | Entrega parcial | Média | Prioridade: aprovar/rejeitar + ativar/inativar funcionando antes dos filtros avançados; filtro básico papel/status/ativo suficiente para o Demo |

## Fora deste sprint (intencional)

- **Integração ViaCEP / auto-preenchimento de endereço** — CEP é preenchido manualmente. Conforto de UX, mas não é caminho crítico.
- **Edição de perfil pelo próprio promotor** após cadastro — Marina não pode corrigir CPF sozinha; deve reenviar a solicitação (US-054) ou criar novo cadastro.
- **Troca de papel pelo admin** (ex.: promover participante comum a promotor sem solicitação formal) — fora do modelo de negócio desta sprint.
- **Exclusão de usuário** — somente inativação. Exclusão tem implicações legais e de rastreabilidade (sprint futura).
- **Tabela de auditoria dedicada (`audit_log`)** — log estruturado nos endpoints admin cobre o rastreio mínimo por ora.
- **Verificação por e-mail do `email_contato`** do promotor — o admin avalia manualmente o dado declarado.
- **Refresh token / logout server-side / blacklist de token** — limitação de sessão ativa pós-inativação é aceita no MVP.
- **2FA / verificação em dois fatores** — fora do roadmap desta sprint.
- **Filtros avançados na tela admin** (ex.: intervalo de datas, busca por CPF) — tabela básica com filtro papel/status/ativo é suficiente para o Demo.
- **Notificação in-app de aprovação/rejeição** — apenas e-mail neste sprint (visível no MailHog em dev).
